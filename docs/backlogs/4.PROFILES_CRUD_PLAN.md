## Storage Architecture

### ObjectBox (current state)

```
~/.eBalistyka/objectbox/
  Owner              ← singleton root (token="local")
  ├── Weapon[]
  ├── Ammo[]
  ├── Sight[]
  ├── Profile[]
  │     ├── ToOne<Weapon>   ← always required (profile without weapon does not exist)
  │     ├── ToOne<Sight>
  │     └── ToOne<Ammo>
  ├── GeneralSettings
  ├── UnitSettings
  ├── TablesSettings
  ├── ConvertorsState
  └── ShootingConditions   ← global session state (not per-profile)

assets/json/collection.json  ← built-in collection (remains JSON)
```

> Weapon is a built-in profile structure logically, but ObjectBox does not support
> embedded objects → one-to-one relation.

## Provider Architecture

### Single Source of Truth — `appStateProvider`

```
ObjectBox streams (Weapon/Ammo/Sight/Profile/Owner boxes)
    │  any write → automatic reload
    ▼
AppStateNotifier  (AsyncNotifier<AppState>)
    │  _load() → AppState { weapons, cartridges, sights, profiles, activeProfile }
    ▼
┌─────────────────────────────────────────────────────┐
│  cartridgesProvider   — List<Ammo>                  │
│  sightsProvider       — List<Sight>                 │
│  weaponsProvider      — List<Weapon>                │
│  activeProfileProvider — Profile?                   │
└─────────────────────────────────────────────────────┘
```

### ShotContext — context for calculation

```
appStateProvider  ──┐
                    ├──► ShotContextNotifier  (AsyncNotifier<ShotContext?>)
shotConditionsProvider ─┘   { profile, conditions }
         │
         ▼
RecalcCoordinator  ← listen(shotContextProvider, settingsProvider, unitSettings)
    │  _triggerAll()
    ▼
homeVmProvider / ShotInfoVMProvider / trajectoryTablesVmProvider
```

### Reactive Settings & Conditions

All settings and condition providers have an ObjectBox stream in `build()`:

```
SettingsNotifier         — watch GeneralSettings box
UnitSettingsNotifier     — watch UnitSettings box
TablesSettingsNotifier   — watch TablesSettings box
ShotConditionsNotifier   — watch ShootingConditions box
```

`save*` methods — DB write only. Stream updates the state itself.

### Profiles Provider Architecture

Three separate providers with clear responsibilities:

```
appStateProvider
    │
    ├──► profilesPagingProvider   Provider<ProfilesPagingState>
    │         Only structural data: orderedIds + activeId.
    │         The active profile is first in the list.
    │         Equality through ==: does not notify if the structure has not changed.
    │         → ProfilesScreen.ref.listen (paging: add/delete/active change)
    │
    ├──► profileCardProvider(id)  Provider.autoDispose.family<ProfileCardData?, String>
    │         Data from one profile: joins weapon/ammo/sight by targetId.
    │         ProfileCardData.== simplified to 7 fields: id, name, weaponFingerprint, ammoId, ammoFingerprint, sightId, sightFingerprint.
    │         → ProfileCard.ref.watch — rebuilds only if this profile has changed.
    │
    └──► profilesActionsProvider  Notifier<void>
              Only actions: selectProfile, removeProfile, createProfile, renameProfile.
              Doesn't hold the position - delegates to appStateProvider.notifier.
```

**Key feature:** synchronous `Provider<T>` has no intermediate states (AsyncLoading/AsyncRefreshing).
`ref.listen` on `profilesPagingProvider` fires only if `ProfilesPagingState.==` returns `false`.
Ammo/sight/weapon content change → `appStateProvider` is updated → `profileCardProvider` is updated →
and `profilesPagingProvider` returns the same `ProfilesPagingState` → paging does not change.

`ProfileCardData` uses three fingerprints instead of checking each field:

| Fingerprint         | What hashes                                                                                                                                                          |
| ------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `weaponFingerprint` | all fields `Weapon` (name, caliberInch, caliberName, twistInch, barrelLengthInch, zeroElevationRad, vendor, notes, image)                                            |
| `ammoFingerprint`   | all fields `Ammo` — display (name, caliber, weight, length, projectileName, vendor) + calc (mv, mvTemp, powderSens, all zero*, bc, dragType, powder/BC tables, etc.) |
| `sightFingerprint`  | all fields `Sight` (name, focalPlane, height, offset, clicks, clickUnits, magnification, reticleImage, calibratedMagnification, vendor, notes, image)                |

`==` simplified to 7 fields: `id`, `name`, `weaponFingerprint`, `ammoId`, `ammoFingerprint`, `sightId`, `sightFingerprint`.
Any change to any entity automatically triggers a card rebuild — without the risk of missing a field.

## Data Ownership

| Data                                                                                         | Belongs to                  | Editable where                        |
| -------------------------------------------------------------------------------------------- | --------------------------- | ------------------------------------- |
| name                                                                                         | Profile                     | ProfilesScreen (on create / rename)   |
| weapon (ToOne)                                                                               | Profile                     | WeaponWizardScreen (from ProfilesScreen) |
| ammo (ToOne)                                                                                 | Profile                     | MyAmmoScreen                          |
| sight (ToOne)                                                                                | Profile                     | MySightsScreen                        |
| caliber, twist, barrelLength, caliberName, vendor                                            | Weapon                      | WeaponWizardScreen                    |
| dragType, bc, mv, mvTemperatureC, zero*, usePowderSensitivity, powderSensitivityFrac, vendor | Ammo                        | AmmoWizardScreen                      |
| sightHeight, vendor                                                                          | Sight                       | SightWizardScreen                     |
| conditions (atmo, wind, distance, powderTemperatureC, usePowderSensitivity)                  | ShootingConditions (global) | ConditionsScreen / HomeScreen         |
| GeneralSettings, UnitSettings                                                                | Owner (global)              | SettingsScreen                        |

> The term "cartridge" and "projectile" remain only in tables, HTML reports
> and assets/json/collection.json. In code — ammo/bullet.

### Ammo — fields and their semantics

| Field                          | Type      | Semantics                                                                                                                  |
| ------------------------------ | --------- | -------------------------------------------------------------------------------------------------------------------------- |
| `muzzleVelocityMps`            | `double?` | MV measured/provided manufacturer                                                                                          |
| `muzzleVelocityTemperatureC`   | `double`  | T₀ is the powder temperature when measuring MV (standard for correction). **Always** passed as `ammo.powderTemp` in bclibc |
| `usePowderSensitivity`         | `bool`    | Enables powder temperature correction                                                                                      |
| `powderSensitivityFrac`        | `double`  | Sensitivity coefficient (fraction / 15°C)                                                                                  |
| `zeroPowderTemperatureC`       | `double`  | Powder temperature at sight-in (if `zeroUseDiffPowderTemperature`)                                                         |
| `zeroUseDiffPowderTemperature` | `bool`    | The temperature of the powder during aiming differs from the atmospheric temperature.                                      |

> `Ammo.powderTemperatureC` and `Ammo.usePowderTempForMv` **removed**. Current
> powder temperature (T₁) is exclusively in `ShootingConditions.powderTemperatureC`.
> `bclibc.Ammo.powderTemp` = T₀ (muzzleVelocityTemperatureC) always;
> `bclibc.Atmo.powderTemp` = T₁ (with conditions or zero) — current.

## Flow Branches

### Flow 1: New Profile

```
ProfilesScreen
  └─ FAB → "Add"
      └─ Bottom sheet: Create new / From collection / Import from file
          └─ Enter profile name + select weapon:
              ├─ "From Collection" → WeaponCollectionScreen
              │     └─ Select → WeaponWizardScreen (pre-filled, caliber readonly)
              └─ "Create manually" → WeaponWizardScreen (empty)
                    └─ Save → appStateProvider.saveWeapon + saveProfile
```

Profile is always created with weapon. Profile without weapon does not exist.
After creation — ammo and sight are not selected.

### Flow 2: Select/Change Ammo

```
ProfileCard (_ProfileControlTile ammo button)
  └─ MyAmmoScreen
      ├─ My Ammo list → Select → appStateProvider.setProfileAmmo → pop
      ├─ "Create Ammo"              → AmmoWizardScreen → saveAmmo
      ├─ "From Collection (cartridge)" → AmmoCollectionScreen (filter: cartridge, caliber filter)
      │       └─ AmmoWizardScreen (pre-filled) → saveAmmo
      └─ "From Collection (bullet)"    → AmmoCollectionScreen (filter: bullet, caliber filter)
              └─ AmmoWizardScreen (pre-filled) → saveAmmo
```

### Flow 3: Select/Change Sight

```
ProfileCard (_ProfileControlTile sight button)
  └─ MySightsScreen
      ├─ My Sights list → Select → appStateProvider.saveProfile
      ├─ "Create Sight" → SightWizardScreen → saveSight
      └─ "From Collection" → SightCollectionScreen
              └─ SightWizardScreen (pre-filled) → saveSight
```

### Flow 4: Edit Weapon

```
ProfileCard
  └─ "Weapon" section tap (ListSectionTile)
      └─ WeaponWizardScreen (pre-filled, caliber readonly)
          └─ Save → appStateProvider.saveWeapon
                  → stream auto-triggers ProfilesViewModel rebuild
```

✅ Implemented. `appStateProvider.saveWeapon` → OB stream → `appStateProvider` reload →
`profileCardProvider(id)` is updated (content) → `profilesPagingProvider` is unchanged (paging).

### Flow 5: Edit Ammo properties

```
ProfileCard
  └─ "Ammo" section tap → onEditAmmo callback → _ProfilesScreenState._onEditAmmo
      └─ reads Ammo + weapon.caliberInch from profile
          └─ AmmoWizardScreen(initial: ammo, caliberInch: weapon.caliberInch)  ← /home/profiles/ammo-edit
              │  extra: (Ammo?, double?) — record transmitted through router
              │  if ammo.caliberInch != weapon.caliberInch → SnackBar with "Update" button
              └─ context.pop(Ammo) → appStateProvider.saveAmmo
```

✅ Routing and callbacks are implemented. AmmoWizardScreen is fully implemented.

### Flow 6: Edit Sight properties

```
ProfileCard
  └─ "Sight" section tap → onEditSight callback → _ProfilesScreenState._onEditSight
      └─ reads Sight entity from appState
          └─ SightWizardScreen(initial: sight)  ← /home/profiles/sight-edit
              └─ context.pop(Sight) → appStateProvider.saveSight
```

✅ Routing and callbacks are implemented, SightWizardScreen is fully implemented.

### Flow 7: Duplicate Profile

```
ProfileCard PopupMenu → "Duplicate"
  └─ Entering a new name
      └─ Copy Profile with new one id
         weapon → copied (new Weapon entity with the same fields)
         ammo   → the same ToOne reference
         sight  → the same ToOne reference
```

✅ Implemented.

## Wizard Screens

### WeaponWizardScreen

Takes `Weapon?` (null = new). Returns `Weapon` via `context.pop(weapon)`.

| Field          | Create   | From Collection | Edit         |
| -------------- | -------- | --------------- | ------------ |
| name           | editable | editable        | editable     |
| vendor         | editable | editable        | editable     |
| caliberName    | editable | editable        | editable     |
| caliber        | editable | **readonly**    | **readonly** |
| twist          | editable | editable        | editable     |
| twistDirection | editable | editable        | editable     | editable |
| barrelLength   | optional | optional        | optional     |

Name field is highlighted in `tertiaryContainer` color + label in `error` color when empty — similar to `NullableUnitValueFieldTile(isRequired: true)`.

### AmmoWizardScreen

Takes `Ammo? initial` (null = new). Returns `Ammo?` via `context.pop(ammo)`.

Used in three contexts:
- `ammo-select/create` — create new (initial = null)
- `ammo-select/cartridge-collection` or `bullet-collection` → AmmoWizardScreen (pre-filled, type readonly)
- `ammo-edit` — edit existing (initial = Ammo with appState)

**Current state:**
- ✅ Name — with touched-validation + immediate required highlighting
- ✅ Vendor — nullable text field
- ✅ Projectile name — nullable, without validation
- ✅ Projectile — Caliber (readonly info), Weight, Length (`NullableUnitValueFieldTile`, isRequired: true)
- ✅ DragModel — `SegmentedButton<DragType>` (G1/G7/CUSTOM) + `_buildBcSection`: Multi-BC switch, Single BC field, "Edit table" ListTile for Multi-BC tables / Custom drag table (with required-highlight if the table is empty)
- ✅ Cartridge — Muzzle Velocity, Muzzle Velocity Temperature (T₀), `usePowderSensitivity` switch
- ✅ Zero Conditions — Distance, Look Angle, Temperature, Pressure, Humidity, Altitude
- ✅ Powder Sensitivity — `PowderSensSection(showToggle: false)` (expanded after switch in Cartridge): diff temp switch, powder temp field, sensitivity input, calculated MV info. Scroll-to-section when enabled. ListTile "Calculate from measurements" → `PowderSensTableEditorScreen`; empty table does not overwrite manually entered coefficient. Algorithm: **pairwise** (between adjacent rows sorted by T), not against wizard reference; 1 valid row → 0; 0 rows → null (not saved).
- ✅ Coriolis — `CoriolisSection`: switch + latitude + azimuth. Scroll-to-section when enabled.
- ✅ Caliber mismatch detection — when editing, `caliberInch` is passed from the weapon profile via record `(Ammo?, double?)`; if there is a difference → SnackBar "Ammo caliber differs from weapon caliber" with the button "Update"
- ✅ Routes to multi-bc editor, custom drag table editor, powder sensitivity table editor — fully implemented

## ProfilesScreen UI

- **FAB** (one `+` button) → bottom sheet with Add flow
- **Per-card actions** → bottom action sheet via ⋮ button (`_ProfileControlTile`): Duplicate, Export, Rename, Remove
- **PageView** with `PageDotsIndicator`
- Active profile — first page (sorted in `profilesPagingProvider`)

**Paging rules** (`ref.listen<ProfilesPagingState>`):

| Event                                         | Action                                                                    |
| --------------------------------------------- | ------------------------------------------------------------------------- |
| Profile added (`length` increased)            | navigate to last page (animated)                                          |
| Profile removed (`length` decreased)          | stay on current page if it still exists; otherwise `clamp` to the nearest |
| Active profile changed (`activeId` different) | navigate to page 0                                                        |
| Content change (ammo/sight/weapon edit)       | nothing — `profilesPagingProvider` does not notify                        |

`ProfileCard` is a `ConsumerStatefulWidget`, watches `profileCardProvider(profileId)` independently.
Each card is rebuilt only if its own data has changed.

## Export / Import

### .ebcp (eBalistyka native format)

JSON-based archive with CBOR packaging. Defined in `packages/ebalistyka_db`:

```
EbcpFile
  version: String        ← kEbcpFormatVersion = '1.0.0'
  items: List<EbcpItem>  ← oneOf: profile / ammo / sight / settings
```

`ProfileExport` — flat DTO with all fields Profile + Weapon + Ammo? + Sight? (without ObjectBox ID).
JSON-serialization via `json_serializable` (`.g.dart`).

Service: `EbcpService` (`lib/core/services/ebcp_service.dart`):
- `shareFile(EbcpFile, name)` — Android/iOS: share_plus; Desktop: FilePicker.saveFile
- `pickAndParse()` — FilePicker → `EbcpFile.fromEbcp(bytes)`
- `buildFullExport(WidgetRef)` — full backup: all profiles + standalone ammo/sight + settings
- `restoreFromExport(EbcpFile, WidgetRef)` — restore via `appStateProvider.notifier`

### .a7p (Archer Ballistic Profile)

Binary protobuff format compatible with A-TACS/Kestrel:

```
[32 bytes ASCII hex MD5][protobuf bytes → proto.Payload]
```

Library: `packages/a7p` (separate Dart package, no Flutter dependencies):
- `A7pFile` — `encode(Payload) → Uint8List` / `decode(Uint8List) → Payload`
- `A7pConverter` — `fromPayload(Payload) → ProfileExport` / `toPayload(ProfileExport) → Payload`
- `A7pValidator` — field validation according to specification (ranges, required fields)
- Proto-schema: `packages/a7p/proto/profedit.proto` (generation: `make proto-a7p`)

Service: `A7pService` (`lib/core/services/a7p_service.dart`):
- `shareFile(ProfileExport)` — Android/iOS: share_plus; Desktop: FilePicker.saveFile
- `pickAndParse()` → `ProfileExport?` — single .a7p picker
- `pickAndParseProfiles()` → `List<ProfileExport>?` — picker with both extensions (.ebcp + .a7p), auto-detect by file extension

### UI (ProfilesScreen)

Per-profile **Export** (⋮ menu → "Export"): bottom sheet with format selection (.ebcp / .a7p).

**Import from file** (FAB → "Add" → "Import from file"): single FilePicker accepts `.ebcp` and `.a7p`, format is determined automatically by extension.

**Settings → Backup**: full backup via `EbcpService.buildFullExport` / `restoreFromExport` (only .ebcp).
