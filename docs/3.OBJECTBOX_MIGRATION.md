# ObjectBox Migration Plan — Simplified Architecture

> **Doc #3 — Status: COMPLETED ✅**  
> All steps done. ObjectBox is the sole storage layer. Archived reference document.

> Phase 6 of the architectural plan. Replaces JSON file storage with ObjectBox, eliminates the conditioning flight, and simplifies provider layers.

---

## Context

The current architecture has too many layers and conditions:
- `JsonFileStorage` → `AppState` (aggregate) → `AppStateNotifier` → `ShotProfileNotifier` → `SettingsNotifier` → `ShotConditionsNotifier` → `ConvertersNotifier`
- Condition flight in `deleteCartridge`/`deleteSight`: state mutates until cascaded storage deletes complete — mismatch on error
- `moveProfileToFirst` saves each profile separately (without transaction)
- ObjectBox (`ebalistyka_db`) is already configured and ready, but not used

**Sollution:**
- ObjectBox = single source of truth for persisted data
- Old classes `WeaponData`/`AmmoData`/`SightData`/`ProfileData` (JSON) → **extension methods on ObjectBox entities**
- `AppStateNotifier` remains the only aggregate in memory (avoiding the condition flight from multiple StreamProviders)
- DB layer is as bare as possible — business logic ONLY in extensions in main app

## New Provider Architecture

**We store `AppStateNotifier` as an aggregate** — splitting into per-entity StreamProviders causes a crash (UI sees an inconsistent state when multiple providers are rebuilt independently).

```
storeProvider   (Provider<Store>)
ownerProvider   (Provider<Owner>)    — singleton Owner (token="local")
    │
    └── appStateProvider (AsyncNotifier<AppState>)
            │  — reads all data from ObjectBox during build()
            │  — CRUD via store.runInTransaction() (atomically)
            │  — AppState: List<Ammo>, List<Sight>, List<Profile>,
            │    GeneralSettings, UnitSettings, TablesSettings,
            │    ConvertorsState, ShootingConditions
            │
            ├── cartridgesProvider      (selector → List<Ammo>)
            ├── sightsProvider          (selector → List<Sight>)
            ├── profilesProvider        (selector → List<Profile>)
            └── validProfilesProvider   (selector)

shotProfileProvider     (AsyncNotifier — watches appStateProvider)
shotConditionsProvider  (AsyncNotifier — watches appStateProvider, reads ShootingConditions)
settingsProvider        (AsyncNotifier — watches appStateProvider, reads GeneralSettings/UnitSettings)
convertorsProvider      (AsyncNotifier — watches appStateProvider)

recalcCoordinator (watches shotProfileProvider + shotConditionsProvider + settingsProvider)
```

**Removed:** `appStorageProvider`, `JsonFileStorage`, `AppStorage` interface

## Race Conditions: How to fix

Old pattern (problematic):
```dart
state = AsyncData(newState); // mutate the state first
await storage.deleteProfile(id); // if fails → mismatch
```

New pattern (ObjectBox transaction):
```dart
store.runInTransaction(TxMode.write, () {
final orphanIds = profileBox
.query(Profile_.ammo.equals(id)).build().find()
.map((p) => p.id).toList();
profileBox.removeMany(orphanIds);
ammoBox.remove(id);
});
// AppStateNotifier rereads data → always consistent
```

## ShotProfileNotifier — Simplification

ObjectBox `ToOne<>` relations are already resolved — no manual `_resolve()` needed:

```dart
class ShotProfileNotifier extends AsyncNotifier<Profile?> {
  Future<Profile?> build() async {
    final appState = await ref.watch(appStateProvider.future);
    return appState.generalSettings.activeProfile.target;
  }

  Future<void> selectProfile(Profile p) async =>
      ref.read(appStateProvider.notifier).saveActiveProfile(p);

  Future<void> selectWeapon(Weapon w) async =>
      ref.read(appStateProvider.notifier).updateProfileWeapon(state.value!, w);

  Future<void> selectSight(Sight s) { ... }
  Future<void> selectAmmo(Ammo a) { ... }
}
```

Removed: `_resolve()`, `_saveCartridgeToGlobalState()`, backward-compat migration.
