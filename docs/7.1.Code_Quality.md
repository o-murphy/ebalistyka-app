## ~~1. Converter Screens — Repeated Structure~~ ✅ DONE

**Problem**: 8 converter screens follow an identical structure but each re-implement the same boilerplate: `_buildInfoTile()`, layout (input picker → divider → info tiles), and provider wiring.

**Applied to**: 5/8 simple screens (length, weight, pressure, temperature, torque).  
**Skipped**: velocity, angular, target_distance — unique layouts, kept as-is.

**What was done**:
- [x] Created `SimpleConvertorScreen` — generic `StatelessWidget` (input picker → sections → info tiles)
- [x] Eliminated `_buildInfoTile()` from all 5 screens — replaced with `InfoListTile` iteration in the generic screen
- [x] Each screen reduced to ~25 lines (thin `ConsumerWidget` wrapper)
- **Actual reduction**: ~270 LOC removed from 5 screens

## ~~2. Converter ViewModels — Repeated Logic~~ ✅ DONE

**Problem**: 8 converter ViewModels share nearly identical structure — watch provider, convert raw value, build display fields, format output.

**Applied to**: 5/8 VMs (length, weight, pressure, temperature, torque).  
**Skipped**: velocity (mach/atmo special cases), angular (two inputs), target_distance (reticle).

**What was done**:
- [x] Created `SimpleConvertorVm` abstract base (Template Method pattern): `build()`, `updateRawValue()`, `changeInputUnit()`, `fieldFor()` helper, `_fmt()` no longer duplicated
- [x] Created `SimpleConvertorUiState` + `ConvertorSection` as shared state types
- [x] Each concrete VM now provides only: `baseUnit`, `getRawBase()`, `getInputUnit()`, `saveRawBase()`, `saveInputUnit()`, `getConstraintsForUnit()`, `buildSections()`
- [x] Provider names unchanged — zero impact on call sites
- **Actual reduction**: ~600 LOC removed from 5 VMs

## 3. Wizard Screens — Duplicated Form Logic 🔴

**Problem**: Three wizard screens share identical patterns — `TextEditingController` lifecycle, `_isValid` getter, `_onSave()` / `_onDiscard()`, `setState` per field, name validation. The `_ActionBar` widget is locally defined in both `WeaponWizardScreen` and `SightWizardScreen` (identical code in two files).

**Affected files**:
- `lib/features/home/sub_screens/weapon_wizard_screen.dart` (286 lines)
- `lib/features/home/sub_screens/sight_wizard_screen.dart` (353 lines)
- `lib/features/home/sub_screens/ammo_wizard_screen.dart` (870 lines)

**What to do**:
- [ ] Extract a shared `WizardActionBar` widget (save/discard buttons) — currently duplicated verbatim
- [ ] Create a `ValidatedNameField` widget — padded TextField with empty-name error state, used in all three wizards
- [ ] Extract a `WizardFormMixin` or base class covering: controller init/dispose, `isNameValid` getter, `_onSave` / `_onDiscard` flow
- [ ] Estimated reduction: ~600 LOC of boilerplate across the three files

## 8. Magic Numbers — Missing UI Constants 🟡

**Problem**: Hard-coded numeric literals are scattered throughout the UI instead of named constants.

**Examples**:
- `EdgeInsets.fromLTRB(16, 8, 16, 8)` — repeated 30+ times
- `Divider(height: 1)` — 37 occurrences
- `Divider(height: 24)` — multiple in converter screens
- Animation `Duration` values — spread across `home_screen.dart`
- `_kMultiBcRowCount = 5`, `_kDragTableRowCount = 100` — local to one file

**What to do**:
- [ ] Create `lib/shared/constants/ui_dimensions.dart` with: `kDefaultPadding`, `kSmallPadding`, `kSectionDividerHeight`, `kTileDividerHeight`, etc.
- [ ] Extract a `SectionDivider` / `TileDivider` const widget for the two divider patterns
- [ ] Move table row count constants to a shared location

## 10. `setState` Overuse in Forms 🟢

**Problem**: Wizard screens manage all form state via `StatefulWidget` + `setState`, calling `setState(() => _field = v)` on every single field change. With Riverpod already in the project, this is inconsistent and slightly wasteful.

**Affected files**: `weapon_wizard_screen.dart`, `sight_wizard_screen.dart`, `ammo_wizard_screen.dart`

**What to do**:
- [ ] Evaluate moving wizard form state to a short-lived `StateNotifier` / `Notifier` per wizard
- [ ] This enables direct validation in the notifier and simplifies the widget tree
- [ ] Low priority — current approach works, this is a consistency improvement

## 11. Naming Inconsistencies 🟢

**Problem**: Private method and variable naming conventions are not uniform across the codebase.

**Examples**:
- Builder methods: some are `Widget _buildXxx()`, some `_buildXxx() => Widget`, some are inline
- Event handlers: some are `_onXxx()`, some are anonymous closures passed directly
- Controller variables: `_xxxCtrl`, `_xxxController` (both used)
- Raw numeric vars: `_xxxRaw`, `_xxxVal`, `_xxx` (all used)

**What to do**:
- [ ] Establish and document conventions in CLAUDE.md:
  - Computed properties → `get xxx`
  - Event handlers → `void _onXxx()`
  - Builder methods → `Widget _buildXxx()`
  - Controllers → `_xxxCtrl`
  - Raw parsed doubles → `_xxxRaw`
- [ ] Apply to new code going forward; fix existing code opportunistically during refactors
